ソフトウェアアーキテクチャ & 設計パターン

本プロジェクト（tetra-backend）のシステム構成、適用されている設計パターン、および各レイヤー・非同期処理等の具体的な実装構造を体系的に解説します。

全体アーキテクチャとレイヤー構造

本システムは AWS SAM (Serverless Application Model) を基盤とし、API Gateway と Lambda を使った完全なサーバーレスアーキテクチャを採用しています。
また、バックエンドプログラムの構造にはクリーンアーキテクチャおよび DDD (ドメイン駆動設計) の考え方を取り入れたレイヤー設計を適用し、インフラ（DynamoDBや外部API等）への依存を切り離した、保守性とテスタビリティの高い設計を実現しています。

各レイヤーの責任境界

  • インターフェース層 (Interfaces - src/interfaces/): API Gateway の HTTP リクエストや SQS トリガーなど外部からの入力を受け取ります。リクエスト検証、Lambda用のレスポンスの組み立て、および共通認可ハンドリングを行います。
  • アプリケーション層 (Application - src/application/): 業務フローを表すユースケースクラスが定義されます。特定のドメインロジックの呼び出し順序や、抽象的なリポジトリインターフェースを介したデータ永続化の調整（オーケストレーション）を担います。
  • ドメイン層 (Domain - src/domain/): 業務のルール（ドメイン知識）そのものを表現する最もコアなレイヤー。エンティティ（例: PointInfo）や値オブジェクト（例: GeoLocation, Email）が含まれ、外部フレームワークやライブラリに一切依存しない設計になっています。
  • インフラストラクチャ層 (Infrastructure - src/infrastructure/): データベース（DynamoDB）、Firebase FCM（プッシュ通知）、SSMパラメータストア、外部のAI APIなど、各種ミドルウェア・外部I/Oとの通信のための具体的な実装を提供します。

主要な適用設計パターン

サーバーレス環境における保守性、拡張性、テスト容易性を確保するために、以下の設計パターンを徹底して適用しています。

依存関係逆転の原則 (DIP) & リポジトリ

ドメイン層でデータベース等の接続仕様を抽象インターフェース（Repository）として定義し、インフラ層で具象（DynamoDB/Firestore）を実装します。これによりビジネスロジックが特定のデータベース技術に直接密結合するのを防ぎます。

ユースケースパターン (単一責任)

CreateChatPointUseCase のように、1つの機能フローごとに1つのユースケースクラスを独立して定義します。コントローラーやハンドラがファットになるのを防ぎ、ユニットテストを各ユースケースごとに容易に作成できるようにしています。

値オブジェクトパターン (Value Object)

GeoLocation（緯度経度検証）や Email、AuthId など、単なるプリミティブ型ではなく独自のイミュータブルなクラスとして実装。値の生成時に強力なバリデーションを実行し、システム全体に不正な値が伝播することを完全に防ぎます。

非同期ワーカー / キューパターン

APIの実行遅延を防ぐために、リアルタイム性が不要な「重い処理（AIを使用したテキスト検閲）」を SQS にキューイングします。API自体は即時レスポンスを返し、後ろの Lambda ワーカーが非同期で処理します。

⚡

非同期コンテンツ検閲（Content Moderation）フロー

スレッド作成時に、AI（DeepSeek API）を用いたコンテンツの有害判定（検閲）を行う仕組みです。 APIの遅延を抑えるため、非同期処理用のキューメッセージング構造が適用されています。

検閲のシーケンス

[Client App]          [API Lambda]            [SQS Queue]            [Moderation Lambda]        [DeepSeek API / DB]
     |                     |                      |                         |                         |
     |-- スレッド作成要求 ->|                      |                         |                         |
     |                     |-- メッセージ送信 -->|                         |                         |
     |                     |   (非同期検閲依頼)   |                         |                         |
     |<-- レスポンス返却 ---|                      |                         |                         |
     |    (即時に作成完了) |                      |-- (SQSポーリング) ---->|                         |
     |                     |                      |                         |-- (AI検閲リクエスト) -->|
     |                     |                      |                         |<-- 有害性チェック判定 --|
     |                     |                      |                         |                         |
     |                     |                      |                         |--- 判定結果を保存 ----->| (DynamoDB: ContentModerationHistories)

非同期化のメリット

AIによるコンテンツ判定には通常数百ミリ秒〜数秒の応答時間を要します。 この判定をスレッド作成処理のHTTP応答サイクルから完全に排除し、SQSを介して非同期ワーカーに処理をバイパスすることで、ユーザーへのレスポンス性能を最大化しています。

Cognito JWT 検証とステートレス認証

バックエンドは各Lambdaに到達するリクエストが認可されたものであるかを判断するため、クライアントが送信した Cognito ID トークン（JWT）の署名検証をステートレスに実行します。

トークン検証の実装パターン（ハンドラ・ユーティリティ）

各APIハンドラは aws-jwt-verify ライブラリを使用し、Cognitoの公開鍵情報をもとに署名、有効期限（exp）、およびクライアントIDを検証し、トークン内の sub クレームから AuthId を取得します。 

import { CognitoJwtVerifier } from "aws-jwt-verify";

export async function getAuthId(event: APIGatewayProxyEvent): Promise<AuthId | null> {
  const authHeader = event.headers.Authorization || event.headers.authorization;
  if (!authHeader || !authHeader.startsWith("Bearer ")) return null;
  
  const token = authHeader.substring(7);
  const verifier = CognitoJwtVerifier.create({
    userPoolId: process.env.COGNITO_USER_POOL_ID!,
    tokenUse: "id",
    clientId: process.env.COGNITO_CLIENT_ID!,
  });
  
  try {
    const payload = await verifier.verify(token);
    return new AuthId(payload.sub);
  } catch (err) {
    logger.error("Token verification failed", err);
    return null;
  }
}
ステートレス認可のメリット

データベースに対する「セッション情報の問い合わせクエリ」を発行することなく、暗号数学的に有効なトークンであることをLambda内で完結して判定できるため、セッションストアの負荷軽減と、Lambda関数のスケール時の極めて低いAPIレイテンシーを実現しています。
